## GITHUB 전략

**브랜치 전략: git flow 기반 + develop 운영**

이번 레포지토리는 GitHub Flow를 기본으로 하되, 팀 협업과 QA를 위해 develop 브랜치를 통합 브랜치로 운영해요.

- 왜 GitHub Flow 기반인가요?
  - 브랜치/규칙이 단순해서 온보딩이 빠르고, 기능 단위로 PR을 올리는 흐름이 명확해요.
  - “작게 만들고 → 빠르게 리뷰받고 → 빠르게 합치기”에 적합해요.

- 왜 develop이 필요한가요?
  - 모든 작업을 바로 main으로 합치면 배포 안정성이 흔들릴 수 있어요.
  - 기능 브랜치들을 먼저 모아 검증하고(리뷰/테스트/스토리북 등), 준비가 되면 main으로 올리는 흐름이 더 안전해요.

**브랜치 역할**

- `main`
  - 배포가 가능한 브랜치예요.
  - 모든 기능이 검증된 후 최종적으로 merge하는 브랜치예요.

- `develop`
  - 기능 브랜치들이 모이는 통합 브랜치예요.
  - 모든 PR은 기본적으로 develop으로 merge해요.

## 브랜치 컨벤션

브랜치는 “무엇을/왜” 하는 작업인지 한눈에 보이도록 type + 기능명 + 이슈 번호로 관리해요.
기능명은 케밥 케이스(kebab-case) 를 사용해요.

- **init/feature-name/#issue-number**
  - 초기 세팅, 환경 구성, 템플릿/룰 셋업 등

- **feat/feature-name/#issue-number**
  - 신규 기능 개발

- **fix/feature-name/#issue-number**
  - 버그 수정

- **refactor/feature-name/#issue-number**
  - 리팩터링(동작 변경 최소화, 구조 개선 중심)

- **Docs/feature-name/#issue-number**
  - 문서 작업 (README, 주석, 가이드, 문서 구조 정리 등)

- **Perf/feature-name/#issue-number**
  - 성능 개선 (렌더링 최적화, 불필요한 연산 제거, 메모리/속도 개선 등)

> merge가 끝난 브랜치는 반드시 삭제해요.

> merge 방식은 Squash Merge를 기본으로 해요.

**왜 이런 규칙을 쓰나요?**

- 브랜치만 봐도 “무슨 작업인지 / 어디 이슈인지” 추적이 쉬워져요.
- Squash merge로 커밋 히스토리를 깔끔하게 유지해서, 나중에 문제 생겼을 때 원인 추적이 빨라져요.
- merge된 브랜치 삭제는 불필요한 브랜치 누적을 막아서 레포 관리가 쉬워져요.

## PR & 커밋 컨벤션

**PR 단위 규칙**

- 하나의 PR에는 최대한 하나의 기능만 포함해요.
  - 예: Header 구현 PR에 Input 컴포넌트 구현까지 같이 넣지 않아요.
- PR이 커지면 리뷰가 느려지고, 리뷰 품질이 떨어지고, 충돌도 늘어나요.
- 그래서 작은 PR을 자주 올리는 방향으로 운영해요.

**PR 제목 규칙**
PR 제목은 Type(모노레포 영역) 로 시작하고, Type은 앞 글자 대문자로 통일해요.

- Feat(client): 로그인 기능 추가

- Feat(cds): Button 컴포넌트 구현

- Init(client): router 설정

- Init(project): PR/Issue 템플릿 세팅

- Fix(client): 메모 저장 시 중복 호출 수정

- Refactor(client): API 에러 처리 구조 개선

**라벨 규칙**

- PR에는 항상 라벨을 달아요. (github-action으로 자동 라벨)
- 라벨은 “리뷰 우선순위/변경 성격/릴리즈 영향도”를 빠르게 판단하게 도와줘요.

**PR 템플릿**

```md
## 📌 Summary

> 관련 있는 Issue를 태그해주세요. (e.g. > - #100)
> _해당 PR에 대한 작업 내용을 요약하여 작성해주세요._

## 📚 Tasks

- _해당 PR에 수행한 작업을 작성해주세요._

## 🔍 Describe

- _해당 PR에서 수행한 작업에 대해서 설명해주세요_

<!--
## 👀 To Reviewer
_리뷰어에게 요청하고 싶은 내용을 작성해주세요._
-->

<!--
## 📸 Screenshot
(기재 내용 없을 경우 섹션 삭제) 작업한 내용에 대한 스크린샷을 첨부해주세요.
-->
```

- 왜 PR 템플릿을 쓰나요?
  - 리뷰어가 무엇을 봐야 하는지 빠르게 파악할 수 있어요.
  - PR의 변경 의도가 정리되면 리뷰 코멘트가 더 정확해지고, 재작업이 줄어들어요.
  - 팀원이 바뀌어도 문서화 방식이 유지돼서 협업 품질이 안정적이에요.

**커밋 컨벤션**

커밋 메시지 형식

> 커밋 메시지는 다음 형식으로 통일해요.

- init: 커밋 메시지

- feat: 커밋 메시지

- fix: 커밋 메시지

- chore: 커밋 메시지

- refactor: 커밋 메시지

- docs: 커밋 메시지

- perf: 커밋 메시지

- test: 커밋 메시지

<br/>

**추가 규칙**

- 특수기호 사용하지 않아요.
- 커밋은 기능별로 가능한 작게 쪼개요.

<br/>

> 왜 커밋을 작게 쪼개나요?

- 문제가 생겼을 때 “어디에서 깨졌는지” 범위를 빠르게 좁힐 수 있어요.
- 리뷰어 입장에서도 변경 흐름을 따라가기 쉬워요.
- squash merge를 하더라도 작업 과정이 깔끔하게 남아요(필요할 때만 확인 가능).

## Issue 컨벤션

Issue 제목은 작업 성격을 바로 알 수 있도록 prefix를 붙여요.

- [Init] 프로젝트 초기 세팅

- [Feat] Tab 컴포넌트 구현

- [Fix] 색상 수정

- [Refactor] API 모듈 구조 개선

또한 Issue 템플릿(feature.yml / fix.yml / refactor.yml)을 사용해서,

- Describe: 무엇을/왜 하는지
- Tasks: 해야 하는 일을 구체적으로
- ETC: 공유할 추가 내용

<br/>

- 왜 Issue 템플릿을 쓰나요?
  - 요구사항이 불명확해서 다시 물어보는 비용을 줄여요.
  - Tasks가 명확하면 PR 스코프가 작아지고, 일정 예측이 쉬워져요.
  - 작업 히스토리(의사결정 근거)가 남아서 나중에 유지보수에 도움돼요.

## 코드리뷰 규칙

**merge 조건**

- PR은 최소 2명 이상의 Approve가 있어야 merge할 수 있어요.
- 직접 push로 합치는 방식은 지양하고, 반드시 PR을 통해 merge해요.

**왜 2명 Approve인가요?**

- 한 명이 놓친 버그/사이드 이펙트를 다른 한 명이 잡을 수 있어요.
- 특정 사람에게 리뷰 부담이 쏠리지 않고, 팀 전체의 코드 이해도가 올라가요.

**리뷰 타임**

- PR이 올라오면 가능한 빨리 리뷰해요.
- 몰아서 한 번에가 아니라, 올라오는 즉시 분산 리뷰가 원칙이에요.

**왜 즉시 리뷰인가요?**

- PR이 오래 쌓이면 충돌이 늘고, 리뷰 난이도도 급격히 올라가요.
- 빠르게 합쳐야 다음 작업이 막히지 않아요.

**리뷰 기준**

- 리뷰 코멘트는 최대한 둥글게 작성해요.
- 근거있는 주장으로 이야기해요
- 가능한 경우 레퍼런스(문서/가이드/팀 규칙)를 함께 첨부해요.

**리뷰에서 중요하게 보는 포인트 예시**

- 아키텍처/폴더 구조 일관성
- 예외 처리/에러 핸들링
- 성능에 영향 줄 수 있는 부분(불필요 렌더링, 무거운 연산 등)
- 타입 안정성(TypeScript), API 계약 준수
- 접근성/UX 고려(로딩/에러 상태 등)
- 네이밍/가독성(우리 팀이 중요하게 보는 기준)

### 네이밍

- **리액트 컴포넌트만 PascalCase**를 사용해요.
  - 예: Header, MemoCard, AiSummaryPanel

- 그 외 유틸/함수/변수는 camelCase를 사용해요.

  <br/>

### export 규칙

- **컴포넌트는 기본적으로 default export**를 사용해요.

- import 경로가 단순해지고, “이 파일의 대표 컴포넌트”가 명확해져요.

- 단, 모듈 집합(index.ts)에서는 named export로 re-export해요.

- 팀에서 import 스타일을 통일하기 위함이에요.

### JSX 작성 규칙

- 의미 없는 div 남발을 지양하고, 가능한 시맨틱 태그를 사용해요.
  - 예: section, header, main, nav, article, aside, footer

- 최상단 래퍼가 의미가 없다면 Fragment(<>...</>)를 사용해요.

- children이 불필요한 컴포넌트는 Self closing을 사용해요.
  - 예: `<EmptyState />`

### 이벤트 핸들러 규칙

- 이벤트 핸들러 함수는 반드시 handle prefix를 사용해요.
  - 예: handleSubmit, handleResetClick

- 이벤트 핸들러가 아닌 일반 함수에는 handle을 사용하지 않아요.
  - “이 함수는 이벤트 핸들러다”라는 의미를 유지하기 위함이에요.

### 네이밍 규칙

- 타입/인터페이스 이름은 PascalCase를 사용해요.

- 기본적으로 type보다 interface를 우선 사용해요.
  - 객체 형태 확장(merge/extends)과 협업 시 확장성 때문에 더 유리해요.

### Props 네이밍

- Props 타입은 반드시 Props 접미사를 붙여요.

- 다음 상황에서는 type을 사용해요.

  - 예: `type ButtonVariant = 'primary' | 'secondary'`

### 타입 파일 구성

- 컴포넌트와 관련된 타입은 가능한 같은 폴더 내부에 co-locate해요.
  - 규모가 커지면 types.ts로 분리해요.

<details>
  <summary>🌎 함수</summary>
  <br/>

### 함수 네이밍(동사 + 명사)

- 함수는 “무엇을 하는지”가 이름만으로 드러나야 해요.

- 의미별 prefix를 통일해요.
  - `get`: 값을 가져와 반환해요. (조회)

  - `create`: 새 값을 생성해요.

  - `check`: 조건을 검사해요.

  - `convert`: 형태를 변환해요.

  - `add / minus`: 더하거나 빼요.

  - `filter`: 필터링 결과를 반환해요.

- boolean 반환 유틸은 is으로 시작해요.
  - 예: isEmailValid, isSubmit

### 함수 선언 방식

- 화살표 함수(arrow function)를 우선 사용해요.
  - 컴포넌트 내부/유틸 함수 스타일을 통일하기 위함이에요.

### key 사용 규칙

- key에 랜덤 값을 넣지 않아요.

- 리스트가 동적이면 반드시 고유 id를 key로 사용해요

- 정적 리스트(순서/개수 변화 없음)에서는 index도 허용해요.
</details>

- 배열 복사는 스프레드 연산자를 사용해요.

- 반복은 가능하면 map, forEach, filter, reduce를 사용해요.

  - Default, WithButtonLabel, Disabled, ErrorState 등

### 테스트 목표

- 테스트는 “모든 걸 다 테스트”가 아니라, 깨지면 치명적인 흐름을 보호하는 게 목적이에요.

### 우선순위

- 유틸 함수 / 변환 로직
  - 입력 → 출력이 명확해서 테스트 효율이 높아요.

- 비즈니스 핵심 플로우
  - 예: 메모 생성/저장/요약 결과 저장 같은 핵심 기능

- 컴포넌트 테스트
  - 상호작용이 중요한 UI (버튼 클릭, 폼 제출, 조건부 렌더링)

### 규칙

- 테스트 파일은 보통 \*.test.ts(x)로 맞춰요.

- 케이스 이름은 “행동/상황/기대 결과”가 명확해야 해요.
  - 예: it('submits form when valid', ...)

</details>

<details>
  <summary>💪 export 규칙 (named rule)</summary>
  <br/>

### 원칙

- 컴포넌트 파일은 default export를 기본으로 해요.
  - 예: ui/button.tsx → export default Button

- 외부로 공개하는 진입점은 index.ts에서 named export로 통일해요.
  - import가 예측 가능해지고, 모듈 경계를 명확히 할 수 있어요.

### 예시

- components/button/ui/button.tsx
  - export default Button;

- components/button/index.ts
  - export { default as Button } from './ui/button';

  - export type { ButtonProps } from './model/types'; (필요 시)

### 왜 이렇게 하나요?

- 컴포넌트 구현 파일에서는 “대표 1개”가 명확해져요.

- 외부에서는 import { Button } from ... 형태로 사용 경험이 통일돼요.

- 추후 파일 구조가 바뀌어도 외부 import를 덜 흔들 수 있어요.

</details>
